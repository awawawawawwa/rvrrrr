<objective>
Refactor the Rust renderer from a synchronous stdin-only read loop to an async tokio-based bidirectional IPC process that reads widget frames from stdin, reads terminal input events via crossterm, and writes lifecycle messages (ready, rendered) and input events to stdout as NDJSON.

Purpose: The Rust side must become a proper IPC participant — responding to lifecycle messages, forwarding input events, and handling EOF/shutdown for crash-safe behavior. This is the foundation for BRDG-02 (stdin EOF detection) and PROT-05 (input event forwarding).

Output: Async Rust binary that speaks the full bidirectional NDJSON protocol.
</objective>

<task type="auto">
  <name>Task 2: Refactor main.rs to async tokio with bidirectional IPC</name>
  <files>crates/renderer/src/main.rs</files>
  <action>
Replace the synchronous `BufRead::lines()` loop in `main.rs` with an async tokio main loop using `tokio::select!` to multiplex stdin reading and crossterm terminal events.

1. Change `fn main()` to `#[tokio::main] async fn main()`:
   - Keep `terminal::install_panic_hook()` at the top.
   - Keep `Terminal::init()` for raw mode + alt screen.
   - Remove the `ctrlc::set_handler` — crossterm event reading in async mode handles Ctrl+C as a key event, and the panic hook + Terminal Drop handles cleanup.

2. After terminal init, write `OutMessage::Ready` to stdout as NDJSON:
   - Use `tokio::io::stdout()` wrapped in `BufWriter`.
   - Helper `async fn write_ndjson(writer, msg)`: `serde_json::to_string(&msg)? + "\n"`, then `writer.write_all()` + `writer.flush().await`.

3. Create the main async loop with `tokio::select!`:
   - **Branch 1 — stdin lines**: Use `tokio::io::BufReader::new(tokio::io::stdin()).lines()` (from `tokio::io::AsyncBufReadExt`).
     - On `Ok(Some(line))`: parse as `InMessage`, handle:
       - `InMessage::Render { root, frame_id }`: paint_tree + compute_diff + emit_diff + swap buffers + write `OutMessage::Rendered { frame_id }` to stdout. Note: painting and diff emission use the existing synchronous crossterm QueueableCommand API on the terminal writer — this is fine, wrap in `tokio::task::spawn_blocking` if needed, but likely the sync IO is fast enough to keep inline (crossterm writes are just buffered syscalls). If spawn_blocking feels heavy, just call the sync render functions directly — they complete in microseconds for typical frames.
       - `InMessage::Resize { width, height }`: resize both buffers.
       - `InMessage::Shutdown`: break the loop.
       - `InMessage::Error { .. }`: log to stderr, continue.
     - On `Ok(None)` (EOF): break the loop — parent is gone (BRDG-02).
     - On `Err(e)`: log to stderr, break.
   - **Branch 2 — crossterm events**: Use `crossterm::event::EventStream` (requires `crossterm` "event-stream" feature). Call `.next().await` on the stream.
     - On `Some(Ok(event))`: call `input::map_crossterm_event(event)`. If `Some(input_event)`, write `OutMessage::Input { event: input_event }` to stdout.
     - On `Some(Err(e))`: log to stderr, continue.
     - On `None`: stream ended, break.

4. After the loop, call `term.cleanup()` (Terminal Drop also handles this, but explicit is clearer).

5. Update `Cargo.toml`:
   - Add `"event-stream"` to crossterm features: `crossterm = { version = "0.29", features = ["event-stream"] }`.
   - Add `futures-core = "0.3"` or just `futures = "0.3"` for the `StreamExt` trait needed by `EventStream.next()`. Actually crossterm's EventStream implements `futures_core::Stream`, so you need `futures = "0.3"` for `StreamExt::next()`. Alternatively use `tokio-stream` — pick whichever is simpler. `futures = "0.3"` is standard.
   - Remove `ctrlc` dependency (no longer needed — crossterm handles Ctrl+C as Event::Key).

6. Important: The existing render pipeline (`paint_tree`, `compute_diff`, `emit_diff`) uses synchronous `crossterm::QueueableCommand` writes to a `BufWriter<Stdout>`. The tokio stdout and the crossterm stdout are separate handles. Approach: Keep the `Terminal` struct's sync `BufWriter<Stdout>` for rendering (crossterm writes), and use a separate `tokio::io::BufWriter<tokio::io::Stdout>` for NDJSON protocol messages. This avoids mixing async and sync IO on the same handle. The crossterm rendering writes go to the terminal's raw stdout, while the NDJSON protocol writes go to tokio's stdout — since this is a child process, stdout is a pipe (not the terminal). WAIT — this is a critical architectural point. When the Rust process is spawned as a child:
   - **stdout** = pipe back to JS (for NDJSON messages)
   - **stderr** = inherited (for logs/backtraces)
   - **terminal** = the actual TTY device

   So crossterm should write to `/dev/tty` (or the platform equivalent), NOT to stdout. The Terminal struct needs to be updated to write to the TTY directly, not stdout. However, modifying Terminal is a significant change. For now, use `crossterm::execute!` with a direct file handle to `/dev/tty` on Unix. On Windows, use `conout$`. Actually, crossterm internally detects the terminal — but its `stdout()` based writes would go to the pipe, not the terminal.

   **Revised approach**: Modify `Terminal::init()` to open the TTY device directly:
   - On Unix: `std::fs::OpenOptions::new().write(true).open("/dev/tty")`
   - On Windows: `std::fs::OpenOptions::new().write(true).open("CONOUT$")`
   - Store this as the writer instead of `BufWriter<Stdout>`
   - All crossterm rendering commands (cursor moves, style, characters) go through this TTY writer
   - stdout remains free for NDJSON protocol messages via tokio

   Update `terminal.rs` to use a TTY file handle instead of stdout. Change `writer` from `BufWriter<Stdout>` to `BufWriter<std::fs::File>`. Update `install_panic_hook` to also use the TTY handle for cleanup.

   Similarly, crossterm's `event::EventStream` reads from the terminal (stdin of the TTY), not from the piped stdin. This should work correctly because crossterm uses platform-specific APIs to read terminal events (on Unix it reads from `/dev/tty`, on Windows from console input).

   Actually, let me reconsider. Crossterm's raw mode and event reading use the process's actual terminal, not stdin. When the process is spawned with `stdio: ["pipe", "pipe", "inherit"]`, the process still has a controlling terminal. Crossterm's `enable_raw_mode()` operates on the terminal, and `event::read()` reads from the terminal input, not from stdin. So crossterm events should work correctly.

   BUT crossterm's `execute!` macro and `QueueableCommand` write to whatever `Write` impl you pass. Currently the code passes `stdout()`. When stdout is a pipe, those ANSI sequences would go into the pipe and corrupt the NDJSON stream. So we MUST change the rendering writer to use the TTY device.

   **Final approach for this task:**
   - Modify `terminal.rs`: Change writer to open TTY device (`/dev/tty` on Unix, `CONOUT$` on Windows). Use `cfg(unix)` / `cfg(windows)` for the path. Wrap in `BufWriter<std::fs::File>`.
   - Update `Terminal::writer()` return type from `&mut BufWriter<Stdout>` to `&mut BufWriter<std::fs::File>`.
   - Update panic hook to also use TTY device for cleanup writes.
   - In `main.rs`, use `tokio::io::stdout()` exclusively for NDJSON output (protocol messages).
